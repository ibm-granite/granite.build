## Overview

Granite.build pushes build events to a RabbitMQ topic exchange in real time.
Instead of polling the API for status changes, you subscribe to a build's events
and receive them as they happen via an AMQP connection.

Key features:
- **Real-time** — events delivered as they occur (sub-second latency)
- **Scoped access** — each subscription gets credentials that can only read events for the requested build
- **No infrastructure to run** — gbserver provisions everything; you just connect
- **Non-blocking** — event publishing never affects build execution

Use cases:
- Live CLI monitoring (watch build progress in terminal)
- Dashboard integrations (real-time status updates)
- CI/CD pipelines (trigger downstream jobs on build success)
- Local notifications (get a ping on your phone when a build finishes)

## Subscribing to Events

### Step 1: Request credentials

Call the subscribe endpoint with your build ID:

```bash
curl -X POST https://gbserver/api/v1/builds/{build_id}/events/subscribe \
  -H "Authorization: Bearer $TOKEN"
```

Response:

```json
{
  "rabbitmq_host": "rabbitmq.internal",
  "rabbitmq_port": 5671,
  "username": "tmp-build-abc12345-xK9f",
  "password": "...",
  "exchange": "build-events",
  "routing_key": "build.abc12345-full-uuid.#",
  "queue": "events.abc12345-full-uuid.xK9f",
  "expires_at": "2026-06-02T18:30:00+00:00"
}
```

### Step 2: Connect to RabbitMQ

Use the returned credentials to connect and consume events. The credentials
are short-lived (60 seconds by default) — connect promptly after receiving them.
Once connected, the AMQP connection persists regardless of credential expiry.

```python
import aio_pika
import json

# Connect with the scoped credentials
connection = await aio_pika.connect_robust(
    host=subscribe_info["rabbitmq_host"],
    port=subscribe_info["rabbitmq_port"],
    login=subscribe_info["username"],
    password=subscribe_info["password"],
)

channel = await connection.channel()

# Declare a temporary queue and bind to the routing key
queue = await channel.declare_queue(exclusive=True, auto_delete=True)
exchange = await channel.get_exchange(subscribe_info["exchange"])
await queue.bind(exchange, routing_key=subscribe_info["routing_key"])

# Consume events
async with queue.iterator() as queue_iter:
    async for message in queue_iter:
        async with message.process():
            event = json.loads(message.body)
            print(f"[{event['event_type']}] {event.get('status', '')} - {event.get('message', '')}")
```

### Access Control

| Caller | Can subscribe to |
|--------|-----------------|
| Build owner | Their own builds |
| Space member | Any build in their space |
| Admin | Any build |

Unauthenticated requests receive `401`. Unauthorized requests receive `403`.

### Reconnection

If your connection drops, call the subscribe endpoint again to get fresh
credentials and reconnect. Events published during the disconnection gap are
not recoverable (the temporary queue is deleted on disconnect).

## Event Payload

Each event is a JSON object published to the exchange:

```json
{
  "build_id": "abc12345-full-uuid",
  "event_type": "status_event",
  "timestamp": "2026-06-02T10:31:12+00:00",
  "target_name": "training",
  "step_name": "space://steps/sft",
  "source": "build-framework",
  "status": "running",
  "message": "Step started"
}
```

| Field | Always present | Description |
|-------|---------------|-------------|
| `build_id` | Yes | UUID of the build |
| `event_type` | Yes | Type of event (see below) |
| `timestamp` | Yes | ISO 8601 timestamp |
| `target_name` | Yes | Target that produced the event (empty for build-level) |
| `step_name` | Yes | Step URI (empty for target-level) |
| `source` | Yes | Event source identifier |
| `status` | Only for `status_event` | Status value (pending, running, success, failed, cancelled) |
| `message` | Only for `status_event` | Human-readable message |

## Event Types

| Event Type | Description | When |
|------------|-------------|------|
| `status_event` | Build/target/step status change | Status transitions |
| `message_event` | Log message from build execution | Build produces output |
| `artifact_event` | Artifact created | New artifact registered |
| `artifact_pushed_event` | Artifact pushed to registry | Artifact uploaded |
| `metrics_event` | Training metrics published | Metrics reported |
| `workload_status_event` | Workload state change | K8s pod state changes |

### Status Values

| Status | Meaning |
|--------|---------|
| `pending` | Build accepted, waiting for resources |
| `running` | Build actively executing |
| `success` | Build completed successfully |
| `failed` | Build failed |
| `cancelled` | Build was cancelled |

### Scope Identification

| `target_name` | `step_name` | Scope |
|---------------|-------------|-------|
| empty | empty | Build-level event |
| set | empty | Target-level event |
| set | set | Step-level event |

## Standalone Notifications

In standalone mode (local builds without RabbitMQ), you can receive
notifications directly on your machine or phone via platform-native adapters.

### Configuration

Create `~/.gbserver/notifications.yaml`:

```yaml
notifications:
  # macOS notification (shows in Notification Center, forwards to iPhone)
  - type: macos
    events: [status_event]

  # Email notification
  - type: email
    to: "you@example.com"
    smtp_host: "smtp.gmail.com"
    smtp_port: 587
    smtp_user_env: "SMTP_USER"        # reads from environment variable
    smtp_password_env: "SMTP_PASSWORD"  # reads from environment variable
    events: [status_event]
```

### Available Adapters

| Adapter | Config type | Requirements | Notifications appear on |
|---------|-------------|--------------|------------------------|
| macOS | `macos` | macOS only | Mac Notification Center + iPhone (via Handoff) |
| Email | `email` | SMTP server | Any phone with email push enabled |

### Event Filtering

The `events` field controls which event types trigger a notification:

```yaml
# Only failures
events: [status_event]

# All events (verbose)
events: ["*"]
```

For `status_event`, notifications are sent on every status change. If you only
want to be notified on completion/failure, the adapter delivers all status
events — filter on the client side or wait for a future `completed`/`failed`
filter option.

### macOS Adapter

Zero setup. Notifications appear in your Mac's Notification Center and can
forward to your iPhone if you're signed in with the same Apple ID and have
Handoff enabled.

```yaml
notifications:
  - type: macos
    sound: default    # optional, plays notification sound
    events: [status_event]
```

### Email Adapter

Sends email via SMTP. Works with Gmail, Outlook, SendGrid, or any SMTP server.

```yaml
notifications:
  - type: email
    to: "you@example.com"
    smtp_host: "smtp.gmail.com"
    smtp_port: 587
    smtp_user_env: "SMTP_USER"
    smtp_password_env: "SMTP_PASSWORD"
    from_addr: "gbserver@example.com"  # optional
    use_tls: true                       # optional, default true
    events: [status_event]
```

Secrets are referenced via environment variables (keys ending in `_env`) —
never stored in the YAML file directly.

## Client SDK

For programmatic use, the `subscribe(build_id, callback)` pattern handles
authentication, connection, and reconnection transparently:

```python
from gbclient import subscribe

async def on_event(event):
    if event.get("status") == "failed":
        print(f"BUILD FAILED: {event['build_id']}")
    elif event.get("status") == "success":
        print(f"Build complete: {event['build_id']}")

# Handles: authenticate → get credentials → connect → consume → reconnect
await subscribe(build_id="abc123", callback=on_event)
```

## Best Practices

**Connect promptly.** Credentials expire in 60 seconds. Call the subscribe
endpoint and connect to RabbitMQ within that window.

**Use temporary queues.** Declare your queue as `exclusive=True, auto_delete=True`
so it's cleaned up automatically when you disconnect.

**Handle reconnection.** If your connection drops, call the subscribe endpoint
again for fresh credentials. Don't cache old credentials.

**Filter by routing key.** If you only need status events, bind with
`build.<id>.status_event` instead of `build.<id>.#` to reduce noise.

**Don't poll.** Events are pushed in real-time. There's no need to poll the
builds API for status — subscribe once and let events flow.

**For long-running monitoring** (dashboards), use a durable named queue instead
of an exclusive queue. This survives brief disconnections without losing events.
