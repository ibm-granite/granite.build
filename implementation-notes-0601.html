Implementation Notes — Fix #60: Skip wandb in standalone mode

Date: 2026-06-01
Branch: fix/standalone-skip-wandb
Issue: #60

Design Decisions

Provider value "none" as the opt-out mechanism
The spec says "skip wandb in standalone mode." Rather than hard-coding a standalone check at every call site, we use the existing GBSERVER_LINEAGE_PROVIDER env var and default it to "none" in standalone mode. This keeps the opt-out explicit and reusable — any user in any environment can set GBSERVER_LINEAGE_PROVIDER=none to disable lineage without needing standalone mode.
Noop class placement
NoopLineageService is in openlineage_service.py (next to the abstract base and factory — small enough to keep inline). NoopLineageStore lives in its own file noop_jobstats.py, mirroring the pattern of wandb_jobstats.py as a separate backend implementation file.

Deviations

Replaced raw os.getenv("GB_ENVIRONMENT", "").upper() == "STANDALONE" with is_standalone()
Applied to constants.py, cloud_logquery_server.py, and cloud_logquery.py. These were pre-existing occurrences unrelated to the wandb fix, but cleaned up for consistency since gbcommon.types.gbenvconfig.is_standalone() already exists and is used throughout gbcli. Removed now-unused import os from the two logquery files.

Tradeoffs

NoopLineageService in-module vs. separate file
Chosen: NoopLineageService stays in openlineage_service.py since it's purely method stubs with no imports beyond what's already there. NoopLineageStore gets its own noop_jobstats.py (per user preference and symmetry with wandb_jobstats.py).
Guarding at factory/accessor level vs. guarding at each call site
Chosen: factory/accessor returns noop. Alternative: wrap each get_lineage_store() call in if not standalone. Rejected because it scatters the concern across buildrunner.py, api/lineage.py, and api/artifacts.py, and would need updating every time a new call site appears.
setdefault in STANDALONE block vs. conditional default in the constant definition
Chosen: setdefault in the existing STANDALONE block at the top of constants.py. This is consistent with how METADATA_STORAGE and BUILDRUNNER_TYPE are already defaulted for standalone. A user who explicitly sets GBSERVER_LINEAGE_PROVIDER=wandb in standalone mode still gets wandb (they must then supply the API key).

Open Questions

Should the lineage REST API endpoints return 503/404 when provider is "none"?
Currently the noop service returns empty results (0 counts, empty search, null graphs). An alternative would be to raise an HTTP error indicating lineage is not configured. I went with silent noops for consistency with the buildrunner path (which already catches and logs exceptions). Let me know if you'd prefer explicit errors on the API endpoints.
